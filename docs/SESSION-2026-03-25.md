## Current Architecture

### Database Schema (post-migration)
All league-scoped tables use `league_key VARCHAR(255)` (e.g., `469.l.4136`).
Personal tables also have `user_id INT` for multi-user isolation.

| Table | Scope | Key Columns |
|-------|-------|-------------|
| `users` | Global | `yahoo_guid`, `email`, `access_token`, `refresh_token` |
| `user_leagues` | Per user | `user_id`, `league_key`, `team_key`, `team_name` |
| `user_selected_league` | Per user | `user_id`, `league_key` |
| `draft_picks` | Per league (shared) | `league_key`, `round`, `pick`, `player_name` |
| `team_rosters` | Per league (shared) | `league_key`, `team_key`, `player_name` |
| `standings` | Per league (shared) | `league_key`, `team_key`, `rank` |
| `matchups` | Per league (shared) | `league_key`, `week`, `team1_key`, `team2_key` |
| `watchlist` | Per user + league | `league_key`, `user_id`, `player_name`, `sort_order` |
| `chat_history` | Per user + league | `league_key`, `user_id`, `prompt`, `raw_response` |
| `player_notes` | Per user + league | `league_key`, `user_id`, `player_name_normalized` |
| `yahoo_player_news` | Per league (cache) | `league_key`, `player_name_normalized` |
| `player_stats` | Global (by season/sport) | `player_name_normalized`, `season`, `sport` |

### API Query Param Convention
All league-scoped endpoints accept `?leagueKey=469.l.4136` (string, not integer).
Personal endpoints use session to determine `user_id` automatically.

### Frontend Data Flow
1. Landing page (`/`) → user picks a league → stored in `localStorage` as full league object
2. Draft room reads `selectedLeague.league_key` from localStorage
3. Chat page reads `league_key` from localStorage, passes to APIs
4. All API calls use `leagueKey` query param or request body field
